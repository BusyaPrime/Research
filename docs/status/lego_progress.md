# LEGO progress по backlog

## Зачем этот файл

Backlog в проекте большой, и без короткого слоя интерпретации он быстро превращается в музей задач. Этот файл нужен, чтобы по эпикам было видно, где мы реально закрыли основу, а где пока живем на честном `mostly_completed`.

## Что видно по backlog сейчас

### Эпики `E01..E19`

Это основа системы, и она уже закрыта достаточно плотно:

- bootstrap и config layer собраны;
- data, PIT, universe, labels, features и gold path работают;
- split engine, baselines и OOF уже не декоративные.

### Эпики `E20`, `E22`, `E24`, `E25`, `E26`

Вот здесь и сидит текущий хвост:

- `E20` — operational runtime еще не тянет advanced rankers;
- `E22` — optimizer в портфельном слое пока упрощенный;
- `E24` — capacity слой рабочий, но его presentation layer еще можно усилить;
- `E25` — report bundle уже хороший, но figures пока в индексе, а не в виде отдельных файлов;
- `E26` — CI и тесты на месте, но до совсем безоговорочного reviewer handoff еще остается несколько шагов.

## Итог

LEGO-метод по факту применяется: проект раскладывается по эпикам, фазам, артефактам и тестам. Но и тут важно не врать себе: backlog уже не “в процессе придумывания”, но и не весь перечеркнут жирной ручкой.
