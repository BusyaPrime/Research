# Governance

Этот каталог нужен для вещей, которые часто считают “не кодом”, а потом именно из-за них проект начинает расползаться. Здесь лежат рабочие артефакты про инварианты, роли, критерии готовности, риски и то, как вообще ревьюить систему так, чтобы она не подглядывала в будущее и не теряла provenance по дороге.

Если коротко: этот слой нужен, чтобы инженер, исследователь и reviewer не держали половину важных правил в голове. Когда правила живут только в голове, они почему-то особенно любят исчезать ровно перед релизом.

## Что здесь лежит

- `goals_and_business_questions.md` — зачем платформа строится и какие вопросы она должна помогать отвечать.
- `unresolved_risks.md` — открытые риски и ограничения, которые нельзя замалчивать.
- `raci.md` — кто за что отвечает в исследовательском контуре.
- `phase_definition_of_done.md` — DoD по фазам.
- `naming_and_versioning.md` — naming conventions и правила versioning.
- `experiment_and_rollback_policy.md` — как запускать эксперименты и как откатываться без лишней драмы.
- `raw_data_immutability.md` — политика по raw layer.
- `anti_leakage_review_checklist.md` — короткий, но злой checklist на тему leakage.
- `demo_plan.md` — как показывать результат так, чтобы это было похоже на инженерную систему, а не на счастливый скриншот.
- `glossary.md` — словарь терминов, чтобы одинаковые слова не означали три разных вещи.

## Как это встроено в систему

Эти артефакты опираются на:

- [MASTER_SPEC.md](/E:/projecttype/docs/specs/MASTER_SPEC.md)
- [machine_spec.yaml](/E:/projecttype/docs/specs/machine_spec.yaml)
- [implementation_notes.md](/E:/projecttype/docs/implementation_notes.md)
- [release_checklist.md](/E:/projecttype/docs/release_checklist.md)

Их задача простая:

- зафиксировать инварианты до того, как они размажутся по обсуждениям;
- сделать ревью воспроизводимым;
- сократить шанс, что следующий инженер будет разгадывать архитектуру по симптомам.

## Что здесь самое важное

Самые критичные документы:

- `anti_leakage_review_checklist.md`
- `phase_definition_of_done.md`
- `raw_data_immutability.md`

Если эти три слоя не держать в порядке, дальше можно очень долго обсуждать модели, а проблема будет в базовой дисциплине данных и времени.

## Подводные камни

- Эти документы не заменяют код и тесты. Они нужны, чтобы код и тесты не начинали жить каждый по своей версии реальности.
- Если реализация расходится с этими правилами, источником истины остается `MASTER_SPEC.md`, а расхождение должно быть явно зафиксировано.
- Любая `TEMPORARY SIMPLIFICATION` обязана быть отражена здесь или в [implementation_notes.md](/E:/projecttype/docs/implementation_notes.md), иначе это уже не упрощение, а тихий долг.

## Как использовать

- перед крупной фазой смотреть `phase_definition_of_done.md`;
- перед ревью model/backtest path смотреть `anti_leakage_review_checklist.md`;
- перед изменением имен, версий и артефактов смотреть `naming_and_versioning.md`;
- перед демонстрацией результатов открыть `demo_plan.md`.

## Как проверять

Минимальная проверка простая:

- файлы на месте;
- ключевые инварианты не противоречат `MASTER_SPEC.md`;
- временные упрощения отражены явно;
- README и implementation notes не обещают того, чего реально нет в коде.

Отдельный smoke-тест на наличие этих артефактов добавлен в `tests/integration/test_governance_artifacts.py`.

## Что еще не готово

- governance-слой уже достаточный для ревью и развития проекта, но его можно еще плотнее связать с machine-readable status audit;
- RACI и demo plan пока рассчитаны на текущий масштаб проекта, а не на большую команду с отдельными vendor/data ops ролями.
